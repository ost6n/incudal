# AGENTS.md

Guidance for AI coding agents working on Incudal.

## Project Overview

Incudal is an Incus-based LXC/KVM NAT VPS sales, delivery, and management panel.

The repository contains:

- `client/`: Vue 3 + Vite frontend.
- `server/`: Fastify + Prisma backend.
- `agent/`: Go host agent for reporting host state and executing host-side operations.
- `server/prisma/`: Prisma schema and migrations.
- `server/templates/`: install scripts, emails, and generated templates.
- `.github/workflows/`: CI, Docker image, release, and agent release workflows.
- `scripts/`: local setup and verification scripts.

## General Rules

- Read the surrounding code before editing. Prefer existing patterns over introducing new abstractions.
- Keep changes scoped to the requested behavior. Avoid unrelated refactors, formatting churn, or dependency changes.
- Do not commit secrets, `.env`, database dumps, generated build output, logs, or local runtime artifacts.
- Treat `server/prisma/schema.prisma` and `server/prisma/migrations/` as production-sensitive. Any schema change must include a migration.
- Use repository commands instead of inventing new scripts when possible.
- Preserve user changes. If the working tree is dirty, inspect it before editing and do not revert unrelated changes.

## Environment

Required for local development:

- Node.js 20 or newer.
- pnpm `9.14.2`.
- PostgreSQL.
- Redis.
- Go only when working on `agent/`.

Initial install:

```bash
corepack enable
corepack prepare pnpm@9.14.2 --activate
pnpm install
```

Local environment variables are expected in `.env` or service-specific environment files. Do not rely on checked-in development secrets.

## Architecture Boundaries

### Frontend

- Vue views live under `client/src/views/`.
- Reusable UI components live under `client/src/components/`.
- API wrappers live in `client/src/api/index.ts`.
- Shared API-facing types live in `client/src/types/api.ts`.
- Locales live in `client/src/locales/`.

When adding frontend behavior:

- Add or update TypeScript API types together with API calls.
- Add translations for `zh-CN`, `zh-TW`, and `en`.
- Respect existing light/dark theme classes and responsive patterns.
- Ensure compact operational UI stays readable on mobile and desktop.
- Prefer existing button, modal, card, input, toast, and table styles.

### Backend

- Fastify routes live under `server/src/routes/`.
- Database access helpers live under `server/src/db/`.
- Cross-cutting services live under `server/src/services/`.
- Shared validation/security/error helpers live under `server/src/lib/`.
- Prisma schema and migrations live under `server/prisma/`.

When adding backend behavior:

- Use the existing authentication decorators such as `fastify.authenticate`, `fastify.authenticateUser`, and `fastify.authenticateAdmin`.
- Add ownership checks for user-owned and host-owned resources.
- Do not trust client-submitted prices, ownership, limits, roles, or computed totals.
- Put balance changes, quota changes, and logs that must be atomic in a single transaction.
- Add rate limits for sensitive or repeatedly callable mutation endpoints.
- Return structured API errors with existing `ErrorCode` entries when practical.

### Agent

- Host agent code lives under `agent/`.
- Agent version is controlled by `agent/VERSION`.
- Release artifacts are generated by `agent/scripts/build-release.sh` and GitHub Actions.
- Do not commit `agent/dist` or local build artifacts.

## Database Rules

- Prisma is the source of truth for relational schema.
- Add migrations for every schema change.
- Keep Prisma field mappings consistent with existing snake_case database columns.
- Use `Decimal` for money-like persisted values and document whether values are stored in cents or yuan.
- Use `BigInt` for byte counters and serialize them as strings in API responses.
- After schema changes, run:

```bash
pnpm --filter server exec prisma generate
pnpm --filter server type-check
```

## Security Rules

- All private API behavior must require authentication.
- Admin-only behavior must use admin authorization, not only frontend hiding.
- Instance owners may operate only on their own instances unless an explicit host-owner or admin path exists.
- Host owners may operate only on instances hosted on their own hosts.
- Never trust frontend prices or billing calculations for payment-sensitive operations.
- Use transactions for balance deductions, refunds, billing records, resource allocation, and state transitions that must stay consistent.
- Add or preserve rate limiting for login, payment, billing, destructive actions, provisioning actions, and other high-impact mutations.
- Do not expose secrets in logs, API responses, screenshots, commits, or generated files.

## Frontend UX Rules

- Keep operational views dense, readable, and consistent with the existing panel style.
- Avoid adding marketing-style sections to app screens.
- Use existing modal and button classes unless there is a strong reason not to.
- New form controls should have clear labels, stable widths, and mobile-safe wrapping.
- Do not put long explanatory text inside narrow controls.
- Do not introduce text that can overflow buttons, segmented controls, cards, or table cells.
- New UI must support light and dark themes.
- New UI text must go through i18n, except for established technical units such as `MB`, `GB`, or `Mbps`.

## Testing Expectations

Run the smallest useful checks for the files you changed. For broad or risky changes, run broader checks.

Recommended checks:

- Frontend type changes: `pnpm --filter client type-check`.
- Frontend build-affecting changes: `pnpm --filter client build`.
- Backend type changes: `pnpm --filter server type-check`.
- Backend build-affecting changes: `pnpm --filter server build`.
- Prisma changes: `pnpm --filter server exec prisma validate` and `pnpm --filter server exec prisma generate`.
- Agent changes: `go test ./...` from `agent/`.
- Full repository confidence: `pnpm build` or the platform-specific local CI script.

If a command cannot run because of missing local services or environment variables, state that clearly in the final response and explain what was verified instead.

## Git Rules

- Check `git status --short` before staging or committing.
- Stage only intended files unless the user explicitly asks to stage all changes.
- Do not amend, rebase, reset, force-push, or rewrite history unless the user explicitly requests it.
- Use concise Conventional Commit-style titles with a type prefix, such as `feat: add paid traffic reset` or `refactor: package page into detail tabs`.
- Do not commit generated folders such as `client/dist`, `server/dist`, `agent/dist`, `node_modules`, or local database/cache artifacts.

## Release And Deployment Notes

- `docker-compose.yml` is the primary deployment path for the panel, PostgreSQL, and Redis.
- Deployment should use a user-provided `.env`; do not hard-code development domains or secrets.
- Production access should be through a reverse proxy such as Nginx, Caddy, or OpenResty with HTTPS.
- GitHub workflows include CI, Docker image building, general release work, and agent release work. Keep workflow changes scoped and explain release impact.
- Agent binaries are published through GitHub Releases and are not stored directly in the panel repository.

## Final Response Checklist

When finishing a task, summarize:

- What changed.
- Which files or areas were touched.
- Which checks were run.
- Any checks that could not be run.
- Any remaining risk or follow-up that matters.
